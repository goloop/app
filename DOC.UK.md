# app - довідник

`app` - ядро життєвого циклу й композиції: воно володіє послідовністю
старту/зупинки сервісу, щоб ваш `main` її не повторював. Це повний український
довідник. Англійською - [DOC.md](DOC.md).

## Модель

`App` тримає впорядкований список компонентів і два списки хуків. Компонент - це
будь-що з `Name`, неблокуючим `Start` і `Stop`. App робить рівно чотири речі:

1. встановлює signal-aware контекст;
2. виконує start-хуки, тоді стартує компоненти в порядку реєстрації;
3. чекає сигнал, скасування батьківського контексту або фатальну помилку
   компонента;
4. зупиняє компоненти у зворотному порядку, тоді виконує stop-хуки, у межах
   таймауту.

Немає global state: усе - поля створеного вами `*App`.

## Створення й опції

```go
a := app.New("api", opts...)
```

Порожнє ім'я стає `"app"`. Опції:

| Опція | Ефект | Дефолт |
|-------|-------|--------|
| `WithLogger(Logger)` | логування життєвого циклу | no-op |
| `WithShutdownTimeout(d)` | дедлайн усього shutdown | 15s |
| `WithSignals(sigs...)` | сигнали, що запускають shutdown | `SIGINT`, `SIGTERM` |

`Logger` - двометодна підмножина `*slog.Logger`, тож `slog.Default()` підходить
напряму:

```go
type Logger interface {
	Info(msg string, args ...any)
	Error(msg string, args ...any)
}
```

## Реєстрація роботи

```go
a.Use(component)                 // старт у порядку, стоп у зворотному
a.OnStart(func(ctx) error {...}) // перед компонентами, у порядку
a.OnStop(func(ctx) error {...})  // під час shutdown, у зворотному порядку
```

Start-хуки виконуються перед стартом компонентів. Якщо start-хук провалюється,
`Run` перериває запуск, виконує stop-хуки (щоб попередні хуки прибрали за собою)
і повертає помилку. Stop-хуки завжди виконуються під час shutdown, після
зупинки компонентів.

## Запуск і shutdown

```go
err := a.Run(ctx)
```

`Run`:

1. вмикає одноразовий прапорець «running» (`ErrAlreadyRunning` на будь-якому
   наступному виклику - App запускається раз, тож реєструйте все до `Run`);
2. загортає `ctx` через `signal.NotifyContext` для налаштованих сигналів;
3. виконує start-хуки;
4. стартує компоненти в порядку; синхронна помилка `Start` зупиняє вже
   стартовані компоненти у зворотному порядку й повертає об'єднану помилку;
5. чекає на одне з: сигнал/батьківський контекст, або фатальну помилку
   компонента;
6. скасовує run-контекст (щоб воркери побачили скасування), зупиняє компоненти
   у зворотному порядку зі свіжим таймаут-контекстом, виконує stop-хуки й
   повертає об'єднану помилку (`errors.Join`). Кожен `Stop` обмежений таймаутом,
   навіть якщо ігнорує свій контекст, тож застряглий компонент не підвісить
   shutdown. `HTTPServer` примусово закриває з'єднання, якщо м'який `Shutdown`
   перевищив дедлайн.

Чистий shutdown за сигналом повертає `nil`. **Другий** сигнал під час shutdown
форсує негайний вихід процесу (код 130), тож повільна зупинка не блокує
нетерплячого оператора.

## Фатальні помилки

Компонент, чия фонова робота впала після успішного старту, повідомляє про це:

```go
app.Fatal(ctx, fmt.Errorf("consumer: %w", err))
```

`Fatal` шле помилку в `Run`, який починає graceful shutdown і повертає цю
помилку об'єднаною з помилками shutdown. Безпечно викликати з горутини,
запущеної в `Start`; поза `Run` або для nil-помилки - no-op. Вбудовані
`HTTPServer` і `Worker` цим користуються.

## Готові компоненти

### HTTPServer

```go
a.Use(app.HTTPServer("http", &http.Server{Addr: ":8080", Handler: mux}))
```

`Start` синхронно біндить листенер (тож «address already in use» одразу спливає
як помилка старту) і обслуговує в горутині; пізніша помилка serve запускає
graceful shutdown через `Fatal`. `Stop` викликає `srv.Shutdown` із shutdown-
контекстом. Порожній `Addr` стає `:http`.

### Worker

```go
a.Use(app.Worker("ticker", func(ctx context.Context) error {
	for {
		select {
		case <-ctx.Done():
			return nil
		case <-ticker.C:
			// періодична робота
		}
	}
}))
```

`Start` запускає блокуючу функцію з run-контекстом, який скасовується на початку
shutdown; функція має повернутись, коли побачить скасування. Повернення не-nil
помилки до shutdown запускає graceful shutdown. `Stop` чекає завершення функції
в межах дедлайну shutdown.

### Closer

```go
a.Use(app.Closer("db", pool.Close))
```

`Start` - no-op; `Stop` викликає функцію. Це іменований, упорядкований
еквівалент `OnStop`-хука - для ресурсу, який треба закрити в конкретній позиції
порядку зупинки.

## Status і health

App не віддає HTTP і не має поняття «health». Він експонує знімок:

```go
type Status struct {
	Name       string
	Components []ComponentStatus
}
func (s Status) Healthy() bool // false, якщо якийсь компонент у StateFailed
```

`Status()` безпечно викликати конкурентно під час роботи App. Зовнішній
інструмент перетворює його на health-check і монтує хендлери сам:

```go
registry.Check("app", func(ctx context.Context) error {
	if !a.Status().Healthy() {
		return errors.New("компонент впав")
	}
	return nil
})
```

Так `app` і будь-який observability-модуль лишаються незв'язаними: жоден не
імпортує інший.

## Конкурентність

`Run` можна викликати по одному за раз; другий конкурентний виклик повертає
`ErrAlreadyRunning`. `Status` захищений м'ютексом і безпечний для читання з
інших горутин. `Start`/`Stop` компонентів викликаються лише з горутини `Run`.

## Межі

`app` робить: життєвий цикл, упорядкований старт/стоп, graceful shutdown,
агрегацію помилок, логування життєвого циклу, знімок стану.

`app` не робить: DI, парсинг конфігу, роутинг, health-реєстр чи HTTP-хендлери,
supervision/restart. Це - інші пакети або ваш код обв'язки.
